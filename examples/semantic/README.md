# Semantic example

This folder shows a **join plus an LLM step**: two tiny CSVs (papers and citations) are serialized, the engine runs a multi-way join, then **`LLM_MAP`** fills in a `keywords` column. Output is written under **`output/`** (for example `ffx_keywords.csv`).

---

### Before you start

1. Build the project from the repo root so your CMake output directory contains **`table_serializer`** and **`query_eval_exec`** (see the main [README](../../README.md)).
2. Point the scripts at that directory:
   - **Recommended:** `export FFX_BUILD_DIR=/path/to/your/cmake/build`
   - **Or** pass `--build-dir …` to `2_serialize.py` and `4_run_query.py`

`1_normalize.py` does **not** need the build directory.

---

### Run the pipeline

From the repository root:

```bash
cd examples/semantic
export FFX_BUILD_DIR=/path/to/your/cmake/build   # adjust

python3 1_normalize.py
python3 2_serialize.py
python3 3_create_config.py
python3 4_run_query.py
```

By default the LLM backend is a **demo** placeholder. To call a real API, set your API_KEY and do the following:

```bash
python3 4_run_query.py --provider openai --model gpt-4o-mini
```

(`FFX_BUILD_DIR` is still read from the environment if you do not pass `--build-dir`.)

---

### What the folders are

| Folder / file | Role |
|---------------|------|
| `data/` | Raw `papers.csv` and `paper_edges.csv`. |
| `work/` | Normalized CSVs with dense IDs. Regenerated by step 1. |
| `serialized/` | Two tables: `papers/` and `paper_edges/` (binaries + metadata). |
| `query.txt` | Join body only. The Python script wraps it with a full rule head and `LLM_MAP(...)`. |
| `ordering.txt` | Join variable order for the engine. |
| `output/` | Created when you run step 4; contains the result CSV. |

---

### Change the query or LLM settings

- **`query.txt`** and **`ordering.txt`** control the relational part.
- **`4_run_query.py`** adds the LLM sink, prompt text, and flags such as `--provider`, `--model`, and `--output`. Open that file to tweak behavior.

---

### Serialization config (optional reading)

`2_serialize.py` runs `table_serializer` **twice** (one pass per CSV). The **`papers`** table uses several named projections so the same row can appear under different relation names in the rule; the **`paper_edges`** table is a simple two-column binary relation (`src`, `dst`).

Example excerpt for `serialized/papers/column_config.txt`:

```
COLUMN,pid,0,uint64
COLUMN,title,1,string
COLUMN,abstract,2,string
PROJECTION,ATitle,0,1,1:1
PROJECTION,AAbs,0,2,1:1
…
```

The citation table uses projections such as `AB`, `BC`, and `R`. See the generated `serialized/*/column_config.txt` files for the full text.
